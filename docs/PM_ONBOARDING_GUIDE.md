# Part 1: Executive Summary

## What Is Project Reachy?

**Project Reachy** is an emotion recognition system for the **Reachy Mini companion robot**. The robot uses a camera to detect human emotions (happy, sad) and responds with:
- **Spoken responses** generated by an AI language model
- **Physical gestures** (waves, nods, empathy gestures)

### The Big Picture

```
┌─────────────────────────────────────────────────────────────────┐
│                    Project Reachy Overview                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   HUMAN          CAMERA          AI MODEL         ROBOT         │
│     😊    →     📷 Video   →    🧠 Classify  →    🤖 Respond    │
│                                 "Happy 92%"       Wave + Speak  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Why It Matters

1. **Privacy-First**: Raw video never leaves the robot — only emotion labels are transmitted
2. **Local Processing**: All AI runs on-device, no cloud dependency
3. **Empathetic Interaction**: Robot responds appropriately to human emotions
4. **Continuous Learning**: User feedback improves future models

# Part 2: Phase 1 Deep Dive

## What Phase 1 Accomplishes

Phase 1 builds the **foundation** for everything else:

### 1. Web Application
- Users can **upload** existing emotion videos
- Users can **generate** synthetic videos using AI (Luma AI)
- Users can **label** videos as "happy" or "sad"
- Videos move through a **curation pipeline** (temp → dataset → train/test)

### 2. ML Training Pipeline
- **Model**: EfficientNet-B0 (HSEmotion weights)
- **Training Strategy**: Two-phase (frozen backbone → fine-tuning)
- **Output**: Trained model that classifies emotions
- **Tracking**: MLflow records all experiments

### 3. Quality Validation
- Models must pass **Gate A** before deployment
- Statistical analysis compares base vs fine-tuned models
- Automated checks ensure models meet minimum thresholds

## Understanding the Statistical Analysis

### Why Statistical Analysis?

We need to **prove** that our fine-tuned model is actually better than the base model. The statistical analysis answers three questions:

| Question | Method | Answer |
|----------|--------|--------|
| Does each model meet quality standards? | Gate A Metrics | Pass/Fail per model |
| Is the fine-tuned model significantly better? | Stuart-Maxwell Test | Yes/No with p-value |
| Which emotion classes improved? | Paired t-Tests | Per-class breakdown |

### Gate A Metrics (Quality Thresholds)

Every model must pass these thresholds before deployment:

| Metric | Threshold | What It Measures |
|--------|-----------|-----------------|
| **Macro F1** | ≥ 0.84 | Overall accuracy-precision balance |
| **Balanced Accuracy** | ≥ 0.85 | Performance across all classes |
| **Per-class F1** | ≥ 0.75 | No single emotion performs poorly |
| **ECE** | ≤ 0.08 | Model confidence is reliable |
| **Brier Score** | ≤ 0.16 | Probability predictions are accurate |

**PM Translation**: 
- "Our model needs to be right at least 84% of the time (F1)"
- "It can't be biased toward one emotion (Balanced Accuracy)"
- "When it says 90% confident, it should actually be right ~90% of the time (ECE)"

### Model Comparison Statistics

| Test | What It Tells Us | PM Interpretation |
|------|-----------------|-------------------|
| **Stuart-Maxwell** | Did overall prediction patterns change? | "Fine-tuning changed something" |
| **McNemar's Test** | Per-class: did error rates change? | "Happy got better, Sad stayed same" |
| **Cohen's Kappa** | How much do models agree? | "Models agree 75% of time" |
| **Paired t-Tests** | Which classes significantly improved? | "Happy improved (p < 0.05)" |

### Research Findings Summary

From the Phase 1 Statistical Analysis paper:

| Finding | Result | Significance |
|---------|--------|--------------|
| Fine-tuned model passes Gate A | ✅ Yes | Ready for deployment |
| Overall improvement | ✅ Statistically significant | p < 0.05 |
| Happy class | ✅ Improved | Large effect size |
| Sad class | ➖ No significant change | Needs more data |

# Part 4: Agent System Overview

## The 10 Agents

Project Reachy uses **10 automated agents** orchestrated by n8n workflows:

| # | Agent | Responsibility | PM Summary |
|---|-------|---------------|------------|
| 1 | **Ingest** | Receive videos, compute checksums | "Video intake" |
| 2 | **Labeling** | Manage human classification | "Tagging system" |
| 3 | **Promotion** | Move videos through pipeline | "Quality control" |
| 4 | **Reconciler** | Ensure database/filesystem sync | "Data integrity" |
| 5 | **Training Orchestrator** | Trigger model training | "Training automation" |
| 6 | **Evaluation** | Run model validation | "Quality testing" |
| 7 | **Deployment** | Push models to robot | "Releases" |
| 8 | **Privacy** | Enforce data retention | "Compliance" |
| 9 | **Observability** | Collect metrics, alerting | "Monitoring" |
| 10 | **Gesture** | Control robot movements | "Robot actions" |

### Agent Flow Diagram

```
User uploads video
       ↓
   [1. Ingest] → Store in temp/
       ↓
   [2. Labeling] → User classifies emotion
       ↓
   [3. Promotion] → Move to dataset
       ↓
   [4. Reconciler] → Verify consistency
       ↓
   [5. Training] → Fine-tune model
       ↓
   [6. Evaluation] → Check Gate A
       ↓
   [7. Deployment] → Push to robot
       ↓
   [10. Gesture] → Robot responds!
```

# Part 5: Quality Gates

## Three Quality Gates

| Gate | When | Who Approves | Key Metrics |
|------|------|--------------|-------------|
| **Gate A** | After training | Automated | F1 ≥ 0.84, Balanced Acc ≥ 0.85 |
| **Gate B** | On Jetson | Automated | Latency ≤ 120ms, Memory ≤ 2.5GB |
| **Gate C** | User rollout | Human (PM) | Real-world performance |

### Gate Approval Flow

```
Training Complete
       ↓
   [Gate A: Offline Validation]
   Pass? → Continue
   Fail? → Retrain with more data
       ↓
   [Gate B: Robot Validation]
   Pass? → Continue
   Fail? → Optimize model size
       ↓
   [Gate C: User Approval]
   Pass? → Full rollout
   Fail? → Investigate feedback
```
