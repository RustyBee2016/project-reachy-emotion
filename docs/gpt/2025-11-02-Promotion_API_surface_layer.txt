Promotion router issue – what went wrong and how we fixed it
•

The FastAPI app is created with root_path="/api/media" (see create_app).

•

The promotion router used to declare APIRouter(prefix="/api/media") and its
endpoints as "/promote/stage" and "/promote/sample".

•

FastAPI composes the public URL as root_path + router prefix + route path, so the
real path became /api/media/api/media/promote/stage.

•

In tests (and any client) we were calling /api/media/promote/stage, which didn’t
match the doubled prefix, hence every request returned 404.

•

Fix: keep the app’s root_path at /api/media, but change the promotion router prefix
to "/promote" and the route paths to "/stage" / "/sample". Now FastAPI assembles
/api/media/promote/stage and /api/media/promote/sample—the intended URLs.
Once that was adjusted, the ASGI tests passed.

About the pytest warnings
DeprecationWarning: 'HTTP_422_UNPROCESSABLE_ENTITY' is deprecated. Use
'HTTP_422_UNPROCESSABLE_CONTENT' instead.
FastAPI (and Starlette) renamed the 422 constant. We currently use
status.HTTP_422_UNPROCESSABLE_ENTITY in the router. Two options:
1. Update those status codes to status.HTTP_422_UNPROCESSABLE_CONTENT
(preferred, keeps everything modern and silences the warning).
2. Suppress the warning in pytest if we need to stay on the old name temporarily (not
recommended long-term).
