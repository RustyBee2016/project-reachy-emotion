4) Implement core routes (the minimum viable backend)
-----------------------------------------------------

**Why:** This is the contract your UI and agents depend on.

**Routes & behavior:**

*   `POST /ingest`
    *   Body: `{source, sha256, label? (optional), duration, fps, width, height, file_name}`.
    *   Server: moves/copies file into `videos/temp/…`, computes/matches sha256 if needed, generates `thumbs/…`, inserts DB row.
    *   Returns: `VideoRead` with URLs (`/videos/...`, `/thumbs/...`).
    *   Idempotency: header `Idempotency-Key`; on duplicate sha256 returns the existing row.
*   `POST /promote`
    *   Body: `{video_id, target_split: train|test, intended_emotion?}`.
    *   Server: transactional move on FS + DB update; enforces “no label in test” and 50/50 guard (reject if breaking policy).
    *   Logs to `promotion_log`.
*   `POST /label`
    *   Body: `{video_id, label: happy|sad}` (only allowed in `train` or `temp`→`train` flow).
*   `GET /videos/{id}`
    *   Returns the `VideoRead` + current URLs.
*   `GET /stats`
    *   Returns per-split counts, per-class counts, and “ready for training: true|false”.

**DoD:** All 5 endpoints pass request/response validation; end-to-end demo: ingest two clips → promote one to `train`+label, one to `test` (no label) → stats reflect 50/50 logic.
